# Постановка: перенос выпусков при активации акции

Документ описывает план доработки процедуры `ActionActivate`: новый режим, в котором проблемный выпуск перед удалением можно попробовать перенести в другое рекламное окно того же дня.

Связанные документы:

- `docs/business-logic.md` - бизнес-термины и правила активации;
- `ArtvisDB/dbo/Stored Procedures/ActionActivate.sql` - текущая процедура активации;
- `ArtvisDB/dbo/Stored Procedures/IssueTransfer.sql` - существующий пример переноса выпуска между окнами;
- `Client/Classes/ActionOnMassmedia.cs` - C# вызов активации.

---

## Текущая логика

Сейчас `ActionActivate` работает жестко:

1. Собирает неподтвержденные выпуски акции.
2. Проверяет каждый выпуск на возможность активации в его текущем `actualWindowID`.
3. Возвращает пользователю списки успешно активируемых и неактивируемых выпусков.
4. При реальной активации (`@isTestActivate = 0`) удаляет неактивируемые выпуски.
5. Подтверждает успешные выпуски и переводит акцию в активное состояние.

Тестовая активация (`@isTestActivate = 1`) только показывает результат проверки и не меняет базу.

---

## Цель доработки

Добавить режим, в котором при реальной активации проблемный выпуск не удаляется сразу.

Если выпуск не может быть активирован в текущем окне, система должна попробовать найти другое подходящее рекламное окно того же дня, перенести выпуск туда и активировать его уже в новом окне.

Если подходящее окно не найдено, выпуск остается неактивированным и удаляется по старой логике.

---

## Бизнес-правила

### Тестовый режим не меняется

Для `@isTestActivate = 1` новая логика переноса не применяется.

Тестовая активация должна работать как раньше:

- ничего не переносить;
- ничего не удалять;
- ничего не подтверждать;
- показать проблемы так, как они есть в текущем размещении.

### Поиск только в рамках того же дня

Если выпуск стоял в окне среды, искать новое окно можно только в эту же среду.

Переход на другой день запрещен.

### Поиск соседних окон

Поиск выполняется вокруг исходного времени выпуска.

Количество попыток задает глубину поиска:

- 1 попытка - ближайшее окно раньше и ближайшее окно позже;
- 2 попытки - если первая пара не подошла, проверить вторую пару раньше/позже;
- и так далее.

### Выбор среди кандидатов

Если в одной попытке найдено несколько подходящих окон, выбирать нужно менее занятое.

Критерий свободности:

```sql
duration - timeInUseConfirmed - уже_принятые_выпуски_текущей_активации
```

Чем больше свободного времени, тем выше приоритет.

Если свободность одинаковая, выбирать более раннее окно.

### Цена окна

Нужна настройка, можно ли переносить выпуск в окно с другой ценой.

Если перенос в окно с другой ценой запрещен, кандидаты ограничиваются окнами с той же `TariffWindow.price`, что и исходное окно.

Если перенос в окно с другой ценой разрешен, кандидатами могут быть окна с другой ценой. Это может повлиять на стоимость кампании, поэтому при реальном переносе нужно обновить `Issue.tariffPrice`.

### Ошибки на первом этапе не фильтруем

В первой реализации не вводим список "переносимых" и "непереносимых" ошибок.

Даже если причина не лечится переносом, например у ролика отсутствует тип рекламы, система все равно может попытаться найти другое окно и заново выполнить полную проверку.

Это проще и надежнее для первой версии. Если позже появятся проблемы с производительностью, можно оптимизировать и не запускать поиск для заведомо непереносимых ошибок.

### Модули и пакеты

Модуль и пакет нельзя активировать частично.

На первом этапе автоматический перенос применяется только к одиночным выпускам линейных кампаний:

```sql
campaignTypeID = 1
moduleIssueID IS NULL
packModuleIssueID IS NULL
```

Групповой перенос модулей и пакетов - отдельная задача.

## Предлагаемые параметры процедуры

Расширить `ActionActivate` параметрами с default-значениями, чтобы существующие вызовы не сломались:

```sql
@tryTransferFailedIssues bit = 0,
@allowDifferentWindowPrice bit = 0,
@transferAttemptCount int = 0
```

Смысл:

- `@tryTransferFailedIssues` - включить новый режим поиска окна;
- `@allowDifferentWindowPrice` - разрешить перенос в окно с другой ценой;
- `@transferAttemptCount` - глубина поиска соседних окон.

Новая логика включается только при условии:

```sql
@isTestActivate = 0
AND @tryTransferFailedIssues = 1
AND @transferAttemptCount > 0
```

## Изменения в результатах процедуры

Сейчас процедура фактически возвращает:

1. `activated` - успешно активируемые выпуски;
2. `notactivated` - неактивируемые выпуски;
3. `fatal_errors` - блокирующие ошибки.

Нужно добавить отдельный набор:

```text
transferred
```

Итоговые result set'ы в логическом смысле:

1. `activated` - активированы без переноса;
2. `transferred` - перенесены и активированы;
3. `notactivated` - не удалось активировать даже после попыток переноса;
4. `fatal_errors` - блокирующие ошибки всей активации.

Физический порядок возврата из процедуры должен сохранять обратную совместимость с текущим C# caller:

1. `activated`;
2. `notactivated`;
3. `fatal_errors`;
4. `transferred`.

Причина: текущий caller уже ожидает первые три набора данных. Новый набор `transferred` безопаснее добавлять в конец, чтобы старые имена/позиции result set'ов не сместились.

C# caller после доработки может показывать `transferred` пользователю отдельной таблицей в удобном порядке, например между "активированное" и "неактивированное", но SQL-порядок возврата должен оставаться совместимым.

## Технический план реализации

### 1. Расширить табличную переменную `@issue`

Добавить служебные поля:

```sql
rollerID int,
grantorID smallint,
oldWindowId int,
oldIssueDate datetime,
isTransferred bit,
transferStatus nvarchar(64)
```

Назначение:

- `rollerID` и `grantorID` нужны для повторной проверки;
- `oldWindowId` и `oldIssueDate` нужны для реального переноса и `TransferLog`;
- `isTransferred` отделяет обычные успешные выпуски от перенесенных;
- `transferStatus` можно использовать для диагностики внутри процедуры.

### 2. Сохранить первичную проверку как есть

Первичный расчет `statusDescription` должен остаться максимально близким к текущему коду.

Это снижает риск регрессии в старом режиме.

### 3. Добавить фазу планирования переносов

После первичной проверки `@issue`, но до возврата result set'ов, добавить фазу:

```text
для каждого проблемного одиночного линейного выпуска:
  найти кандидатные окна того же дня
  проверить кандидатов по тем же правилам активации
  выбрать лучшее окно
  если окно найдено:
    пометить выпуск как OK
    сохранить новое окно в @issue
    записать план переноса в @plannedTransfers
```

Фаза выполняется только для реальной активации и только при включенном режиме переноса.

### 4. Кандидаты для переноса

Кандидат должен удовлетворять базовым ограничениям:

```sql
tw.massmediaID = исходное massmediaID
tw.dayActual = исходный день
tw.windowID <> исходный windowID
```

Если `@allowDifferentWindowPrice = 0`:

```sql
tw.price = исходная price
```

Соседство считать по расстоянию от исходного времени:

```sql
ABS(DATEDIFF(minute, исходное_время, tw.windowDateActual))
```

Но глубину попыток лучше считать не по минутам, а через ранжирование окон до и после исходного:

```text
rank_before = 1, 2, 3...
rank_after  = 1, 2, 3...
attemptNo = rank_before/rank_after
```

Так `@transferAttemptCount = 2` означает именно две пары соседних окон, даже если интервалы между окнами неравномерные.

### 5. Проверка кандидата

Кандидат должен проходить те же проверки, что обычная активация:

- окно не disabled;
- ролик допустим для типа окна;
- спецпозиция не занята;
- хватает времени или capacity;
- дата допустима;
- deadline не нарушен;
- права `rightForMinus` и `grantorID` учитываются как сейчас;
- у ролика есть тип рекламы.

### 5.1. Позиция выпуска при переносе

При автоматическом переносе `positionID` сохраняется.

Если выпуск был помечен как первый (`positionID = -20`), система ищет окно, где первая позиция свободна. Если такого окна нет в пределах разрешенных попыток, выпуск не переносится.

То же правило применяется ко второй и последней позиции.

Автоматический перенос не должен сбрасывать позицию в обычную (`0`) и не должен сам переводить ее в специальные "transferred"-позиции (`-15`, `-5`, `5`). Такие позиции используются в других сценариях интерактивного переноса и не являются частью базового режима автоспасения при активации.

Важно: при проверке вместимости учитывать не только текущий `TariffWindow.timeInUseConfirmed`, но и уже запланированные к активации выпуски из текущего запуска процедуры.

То есть сохраняется текущая идея:

```sql
tw.timeInUseConfirmed
+ duration_текущего_выпуска
+ sum(duration уже OK выпусков из @issue в это окно)
<= tw.duration
```

### 6. Реальный перенос в базе

После формирования result set'ов, в финальном блоке реальной активации, перед подтверждением выпусков выполнить переносы из `@plannedTransfers`.

Механика переноса:

1. Снять unconfirmed-занятость со старого `actualWindowID`.
2. Обновить `Issue.actualWindowID` на новое окно.
3. Добавить unconfirmed-занятость в новое окно.
4. Пересчитать `Issue.tariffPrice` по цене нового окна.
5. Записать строку в `TransferLog`.

Образец механики есть в `IssueTransfer.sql`.

Прямо вызывать `IssueTransfer` из `ActionActivate` не рекомендуется без отдельной проверки, потому что `IssueTransfer` рассчитан на интерактивный перенос и имеет собственные сообщения/проверки. Безопаснее переиспользовать его алгоритм внутри `ActionActivate`.

### 7. Подтверждение выпусков

После переносов старая фаза подтверждения может остаться почти без изменений:

```sql
update Issue
set isConfirmed = 1, activationDate = getdate()
from Issue i
inner join @Issue it on i.issueID = it.issueID
where it.statusDescription = 'OK'
```

Так перенесенные выпуски активируются вместе с обычными успешными, потому что после успешного планирования переноса они получают `statusDescription = 'OK'`.

### 8. C# изменения

В `ActionOnMassmedia.ActivateAction` нужно:

1. Передавать новые параметры в `ActionActivate`.
2. Обработать новый result set `transferred`.
3. Показать пользователю отдельную таблицу перенесенных выпусков.

Поскольку параметры SQL будут с default-значениями, старый вызов можно сохранить до появления UI-настроек.

## Ценовой блок

Если перенос в окно с другой ценой разрешен, нужно обновлять `Issue.tariffPrice`.

Причина: `ActionRecalculate` для линейных кампаний суммирует сохраненный `Issue.tariffPrice`; он не пересчитывает цену каждого выпуска заново по новому окну.

Значит, при переносе нужно:

1. получить новую цену окна;
2. получить надбавки за позиционирование из прайслиста;
3. пересчитать цену через `fn_GetIssuePrice`;
4. обновить `Issue.tariffPrice`.

---

## Риски

- `ActionActivate` уже сложная процедура; изменения должны быть минимальными и локальными.
- Нужно не сломать тестовую активацию.
- Нужно не сломать текущий жесткий режим.
- Нужно аккуратно учитывать выпуски, которые уже стали `OK` в рамках того же запуска активации.
- Модули и пакеты нельзя переносить частично.
- Перенос в окно с другой ценой без пересчета `Issue.tariffPrice` даст неверную стоимость кампании.

---

## Проверка

Минимальные сценарии проверки:

1. Старый режим: параметры переноса выключены, поведение полностью прежнее.
2. Тестовая активация: параметры переноса включены, но `@isTestActivate = 1`; переносов и изменений в БД нет.
3. Выпуск не проходит из-за переполнения окна, соседнее окно подходит; выпуск попадает в `transferred` и активируется.
4. Выпуск не проходит из-за занятой первой позиции, соседнее окно со свободной первой позицией подходит; выпуск переносится.
5. Подходящих окон нет; выпуск остается в `notactivated` и удаляется по старой логике.
6. Два равно свободных кандидата; выбирается более раннее окно.
7. `@allowDifferentWindowPrice = 0`; окно с другой ценой не используется.
8. `@allowDifferentWindowPrice = 1`; окно с другой ценой может использоваться, но ценовой пересчет должен быть проверен отдельно.
9. Модульный/пакетный выпуск не переносится одиночно.
10. После реальной активации `TariffWindow` counters согласованы: старое окно освобождено по unconfirmed, новое окно занято, затем успешные выпуски переведены в confirmed.
